# [AI 협업 기록] GitHub 인증 시스템 및 UI 개발 공정

본 문서는 **DevFlow** 프로젝트 구현 과정에서 발생한 AI(Gemini)와의 협업 내역을 투명하게 기록한 작업 일지입니다.

## 1. 개요

- **협업 대상**: Gemini 3 Pro (빠른 답변)
- **협업 기간**: 9주차
- **주요 협업 분야**: GitHub REST API (Octokit) 연동, Zero-Exposure 보안 설계, TypeScript 엄격한 타입 핸들링

---

## 2. 상세 협업 로그

### 🚀 [LOG-1] Zero-Exposure 보안 지향적 인증 아키텍처 설계

- **프롬프트 (질문):** "GitHub 푸시 기능을 만들 건데, 클라이언트 세션에 `accessToken`을 노출하면 보안상 위험할 것 같아. 브라우저에서는 토큰이 안 보이면서 서버에서만 안전하게 깃허브 API를 호출하는 방법이 있을까?"
- **AI 제공 해결책:**
- 클라이언트 `useSession` 결과물에서 토큰을 배제하도록 `session` 콜백 수정 가이드.
- 서버 사이드 전용 JWT 추출 함수인 `getToken`을 사용하여 API Route 내부에서만 인증 정보를 취득하는 아키텍처 제시.

- **직접 수행한 작업:**
- `authOptions`의 `session` 콜백에서 토큰 할당 로직을 완전히 제거하여 보안 무결성 확보.
- 서버 API(`/api/github-push`)를 생성하고, 서버 환경에서 직접 쿠키를 읽어 GitHub 통신을 대행하는 대리인(Proxy) 패턴 적용.

---

### 🛠️ [LOG-2] Octokit 기반 GitHub REST API 통신 레이어 구축

- **프롬프트 (질문):** "GitHub에 파일을 푸시할 때 직접 `fetch`를 쓰는 게 좋을까, 아니면 라이브러리를 쓰는 게 좋을까? 용량과 효율성 측면에서 분석해줘."
- **AI 제공 해결책:**
- GitHub 공식 라이브러리인 **`Octokit`** 추천.
- `SHA` 기반의 파일 업데이트 로직과 `Base64` 인코딩 등 복잡한 절차를 표준화된 메서드로 처리할 수 있음을 설명.

- **직접 수행한 작업:**
- `pnpm add octokit`을 통해 라이브러리를 도입하고, 파일 존재 여부에 따라 `sha` 값을 조건부로 할당하여 '신규 생성'과 '덮어쓰기'가 모두 가능한 견고한 푸시 로직 완성.

---

### ⚠️ [LOG-3] TypeScript 엄격한 타입 체크 대응 (Zero-Any)

- **프롬프트 (질문):** "TypeScript 설정이 엄격해서 `error: any`나 `as any`를 쓰면 전부 에러가 나. `unknown` 에러 객체에서 안전하게 `message`나 `status` 속성을 꺼내 쓰는 법을 알려줘."
- **AI 제공 해결책:**
- **커스텀 타입 가드(Type Guard)** 함수인 `isGitHubError`를 구현하여 논리적으로 타입을 좁히는(Narrowing) 기법 제시.
- `instanceof Error`를 활용한 표준 에러 핸들링 가이드.

- **직접 수행한 작업:**
- 프로젝트 내의 모든 `any` 캐스팅을 제거하고, `interface GitHubError`를 정의하여 API 통신 시 발생할 수 있는 모든 예외 상황을 타입 안전하게 처리.

---

### ⚠️ [LOG-4] ID 생성 로직 개선을 통한 데이터 충돌 방지

- **프롬프트 (질문):** "Date.now()를 ID로 쓰면 사용자가 연타할 때 충돌이 날 수 있다는 지적을 받았어. 라이브러리 없이 더 안전한 ID를 만드는 방법이 뭐야?"
- **AI 제공 해결책:**
- 브라우저 표준 API인 **`crypto.randomUUID()`** 사용 권장.

- **직접 수행한 작업:**
- `useDocStore`의 `addDocument` 함수 내 ID 생성 로직을 UUID 방식으로 변경하여, 1ms 내 중복 요청 시에도 데이터가 덮어씌워지지 않는 고유 식별자 체계 구축.

---

### 🔍 [LOG-5] GitHub 유저 식별자(Username) 정밀 매핑 및 404 에러 해결

- **프롬프트 (질문):** "GitHub API를 호출할 때 404 Not Found가 떠. 로그를 찍어보니 `owner`가 프로필 이름인 `Yujin Jeon`으로 들어가는데, 내 진짜 닉네임인 `yuj2n`으로 인식이 안되는 이유가 뭐야? 어떻게 해결해야해"
- **AI 제공 해결책:**
- NextAuth의 `jwt` 콜백에서 깃허브 프로필(`Profile`) 객체의 `login` 필드(실제 ID)를 추출하여 토큰에 저장하는 방식 제시.
- `Display Name`과 `Username`의 차이를 설명하고, API 엔드포인트 구성 시 후자를 사용해야 함을 명시.

- **직접 수행한 작업:**
- `authOptions`의 콜백 함수를 수정하여 깃허브 고유 ID를 세션에 통합.
- 에디터 페이지의 푸시 요청 파라미터를 `session.user.name`에서 새로 정의한 `session.user.username`으로 교체하여 경로 무효화 문제 해결.

---

### 🏗️ [LOG-6] Zustand를 활용한 전역 설정(Configuration) 및 영속화(Persistence)

- **프롬프트 (질문):** "사용자가 선택한 레포지토리랑 저장 경로를 매번 입력받지 않고 저장해두고 싶어. 새로고침해도 안 날아가게 하려면 어떻게 해야 해?"
- **AI 제공 해결책:**
- **Zustand**와 **`persist` 미들웨어**를 결합한 전역 설정 스토어 설계 제안.
- 브라우저의 `localStorage`와 상태를 동기화하여 별도의 DB 없이도 사용자 설정을 유지하는 'Persistence' 기법 가이드.

- **직접 수행한 작업:**
- `useConfigStore`를 생성하여 `selectedRepo`, `targetDir` 등의 설정을 전역 관리.
- 설정 페이지(`GithubConnect`)와 에디터 페이지 간의 데이터 단방향 흐름을 구축하여, 설정 변경 시 에디터의 푸시 동작이 즉시 반영되도록 구현.

---

### 🛡️ [LOG-7] TypeScript Module Augmentation을 통한 세션 타입 확장

- **프롬프트 (질문):** "세션에 `username`을 넣었는데 타입스크립트에서 계속 에러가 나. `as any`를 쓰지 않고 정석대로 이 필드를 인식하게 만드는 방법이 있나?"
- **AI 제공 해결책:**
- **Module Augmentation** 기법을 사용하여 `next-auth` 모듈의 내부 인터페이스(`Session`, `JWT`)를 재정의하는 솔루션 제공.
- `GithubProfile` 인터페이스 캐스팅을 통해 프로필 데이터 접근 시 타입 안정성을 확보하는 법 설명.

- **직접 수행한 작업:**
- `types/next-auth.d.ts` 파일을 생성하여 커스텀 속성에 대한 전역 타입 선언 완료.
- 프로젝트 전체에서 `as any` 캐스팅을 제거하고, 자동 완성(IntelliSense) 기능을 활용한 타입 안전 코딩 환경 구축.

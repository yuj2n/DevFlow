# [AI 협업 기록] 14주차: 관심사 분리 기반 UI 최적화 및 템플릿 동적 주입 시스템 구축

본 문서는 **DevFlow** 프로젝트 구현 과정 중 14주차에 진행된 전역 설정 대시보드 리팩토링, 문서 레이아웃(Template) 수명 주기 싱크 연동, 그리고 Zustand Persist 환경에서 발생한 가스케이딩 렌더링(Cascading Renders) 무한 루프 결함 해결에 관한 AI(Gemini)와의 협업 내역을 투명하게 기록한 작업 일지입니다.

## 1. 개요

- **협업 대상**: Gemini 3 Pro (빠른 답변)
- **협업 기간**: 13~14주차 (최종 고도화 및 안정화 마일스톤)
- **주요 협업 분야**: 관심사 분리(CoC) 기반 UI 구조 정돈, Zustand 영속성 수명 주기 동기화, 동적 템플릿 마크다운 뼈대 엔진 설계

## 2. 상세 협업 로그

### 📐 [LOG-1] 대시보드 피로감 해소를 위한 관심사 분리(Separation of Concerns) UI 구조 개편

- **프롬프트 (질문):** "기획 고도화 차원에서 저장소 설정이랑 일반 설정을 한 페이지로 다 합쳤더니 기능이 너무 몰려서 가독성이 엉망이 됐어. 다시 깔끔하게 분리하면서 환경 설정 페이지 기능들만 완벽하게 연동하는 방법이 있을까?"
- **AI 제공 해결책:**
- 단일 화면에 성격이 다른 애플리케이션 제어(에디터 테마, 알림 웹훅)와 인프라 제어(깃허브 저장소, 브랜치, 디렉토리 경로)가 집중될 때 사용자가 직면하는 시각적 인지 과부하(대시보드 피로감)를 아키텍처 관점에서 진단.
- 서비스 환경 설정은 "가볍고 개인적인 Preferences 공간", 깃허브 연동은 "전문적인 Pipeline 인프라 공간"으로 완벽히 격리하는 **관심사 분리(Separation of Concerns)** 관점의 UI 롤백 가이드라인 및 레이아웃 정돈안 제시.

- **직접 수행한 작업:**
- 무분별하게 중복 렌더링되던 깃허브 계정 연결 블록을 과감히 걷어내고, `DOCUMENT TEMPLATE`, `PREFERENCES`, `NOTIFICATIONS` 3가지 핵심 기능만 한눈에 들어오는 세련된 2단 분할 그리드 디자인 밸런스 회복.
- 전역 스토어(`useConfigStore`)의 원래 목적에 맞게 설정 관리 도구의 저장 범위를 정제하여 UI 가독성과 사용성을 대폭 개선함.

### 🚨 [LOG-2] Zustand Persist 동기화 시 발생하는 가스케이딩 렌더링(Cascading Renders) 결함 제어

- **프롬프트 (질문):** "설정 페이지에서 분명 API Spec 레이아웃을 고르고 저장을 눌렀는데, 페이지를 새로고침하거나 다시 돌아오면 자꾸 첫 번째인 Standard 테마로 불빛이 돌아가 버려. 그리고 콘솔 창에 cascading renders 경고 에러가 뜨는데 어떻게 고쳐야 해?"
- **AI 제공 해결책:**
- Zustand의 `persist` 미들웨어가 브라우저 로컬 스토리지에서 상태를 읽어와 복원(Hydration)하기 전에, React 컴포넌트가 부팅되며 `useState` 초기값으로 동기적 바인딩을 먼저 시도하여 발생하는 **서버-클라이언트 간 상태 불일치(Hydration) 결함** 진단.
- `useEffect` 내부에서 동기적으로 `setState`를 실행 시 렌더링 스케줄링이 충돌해 [화면 렌더링 ➡️ Effect 실행 ➡️ State 변경 ➡️ 재렌더링]으로 이어지는 **가스케이딩 렌더링(Cascading Renders) 무한 루프** 현상 규명 및 방어막 상태(`isHydrated`) 플래그 도입 제안.

- **직접 수행한 작업:**
- `SettingsPage` 컴포넌트 내부에 비비드한 화면 락(Lock) 역할을 하는 `isHydrated` 라이프사이클 방어벽 상태 설계.
- 로컬 스토리지의 온전한 복원이 검증된 마운트 시점에 딱 최초 1회만 `setPendingTemplate`, `setPendingTheme` 등의 내부 상태 주입 연산이 수행되도록 격리벽을 구축하여, **새로고침 및 페이지 이동 후에도 유저 세팅이 휘발되지 않는 고신뢰성 설정 영속화 시스템을 완성**함.

### 📋 [LOG-3] 환경 설정 맞춤형 동적 문서 생성(Document Template Insertion) 시스템 설계

- **프롬프트 (질문):** "유저가 환경 설정에서 고른 레이아웃 설정이 문서함 페이지(`DocumentListPage`)에서 `+ 새 문서 만들기` 버튼을 누를 때 실제 뼈대 데이터로 주입되게 만들고 싶어. 스토어 상태랑 어떻게 엮어야 할까?"
- **AI 제공 해결책:**
- 환경 설정 페이지에서 세이브한 `template` 리터럴 키값(`standard`, `api spec`, `meeting notes`)을 문서 추가 핸들러의 실행 트리거 내부로 수혈하는 데이터 단방향 흐름 파이프라인 설계.
- Tiptap 에디터가 파싱 오류 없이 수용하는 레이아웃별 초기 HTML 상수를 선언하고, 인덱서(`TEMPLATE_CONTENT[template]`)를 활용해 무효한 키 접근을 차단하는 예외 방어벽 구조 가이드 제공.

- **직접 수행한 작업:**
- `DocumentListPage.tsx` 내부 생성 로직에 `useConfigStore` 상태 감지 레이어 안착.
- `+ 새 문서 만들기` 클릭 시 유저 설정 상태에 따라 [Standard: 일반 기술 노트 서식 / API Spec: 요청 파라미터 테이블 및 응답 예시 코드 블록 탑재 서식 / Meeting Notes: 참석자 및 안건 스크럼 서식]의 완전체 뼈대 HTML 데이터가 자동으로 채워진 채로 에디터가 열리는 지능형 템플릿 컨텍스트 구현 성공.
